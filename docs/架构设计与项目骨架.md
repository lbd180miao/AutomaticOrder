# AutomaticOrder 架构设计与项目骨架

## 1. 项目目标

本项目定位为保险杠自动下线、激光打标、机器人空中交接、视觉装箱、泡棉贴附与 MES 追溯的上位机管理系统。

系统采用 Django + Template + SQLite 作为基础技术栈，OpenCV 作为视觉处理核心。Django 不承担 PLC 和机器人毫秒级实时控制职责，而是作为上位机业务中枢，负责流程状态管理、数据记录、视觉结果判定、MES 交互、报警闭环和页面展示。

现场实时动作仍由 PLC、机器人控制器、激光打标机、扫码枪、相机和传感器执行。Django 通过设备适配层与这些设备进行数据交互。

## 2. 需求范围

### 2.1 阶段一：注塑下线与激光打标

注塑机取件机器人完成产品下线、移动至激光打标位、等待 PLC 触发打标。打标完成后，扫码枪读取产品条码，系统将产品条码与料框码上传 MES。打标和数据上传成功后，注塑机器人携带产品移动至空中交接位，并向 PLC 发送待交接信号。

Django 侧关注点：

- 记录产品条码。
- 记录打标结果。
- 记录扫码结果。
- 记录 MES 上传状态。
- 记录产品从下线到待交接的流程状态。
- 在异常时形成报警记录。

### 2.2 阶段二：空中交接

装箱机器人移动至空中交接位，PLC 同时确认注塑机器人待交接和装箱机器人就绪。PLC 下发抓取指令后，装箱机器人通过真空确认产品已抓牢。PLC 再通知注塑机器人释放产品，注塑机器人释放完成后返回注塑机旁。

Django 侧关注点：

- 记录交接握手状态。
- 记录装箱机器人到位状态。
- 记录真空确认结果。
- 记录注塑机器人释放完成状态。
- 记录交接异常、超时和人工处理记录。

### 2.3 阶段三：视觉装箱与泡棉组装

装箱机器人独立执行料架定位、产品装箱和泡棉贴附。视觉系统通过深度相机计算左右料架的 X/Y/Z 偏差，并进行层高、层距与 MES 配方校验。装箱完成后，机器人抓取泡棉并贴附到指定位置，固定检测相机拍照，视觉系统校验泡棉贴附状态。

Django 侧关注点：

- 记录料框码。
- 调用 MES 获取装箱配方。
- 记录视觉采集图片。
- 记录左右料架定位结果。
- 记录三轴补偿数据。
- 记录层高、层距配置比对结果。
- 记录装箱循环进度。
- 记录泡棉贴附检测结果。
- 记录合格、不合格、锁定工位、人工处理状态。

## 3. 技术栈

### 3.1 后端

- Python
- Django
- Django Template
- SQLite

### 3.2 视觉

- OpenCV
- NumPy
- 相机 SDK 适配层，前期可先使用 OpenCV `VideoCapture` 或图片文件模拟。

### 3.3 设备通讯

设备通讯在项目初期只做抽象和模拟，不直接绑定具体厂商协议。

建议预留以下通讯方式：

- PLC：Modbus TCP、OPC UA、Socket 或厂商 SDK。
- 机器人：PLC 中转、Socket、文件接口或厂商 SDK。
- 扫码枪：串口、TCP 或 PLC 转发。
- 激光打标机：PLC 触发或打标机 SDK。
- MES：HTTP API、WebService、数据库中间表或文件接口。
- 相机：OpenCV、GigE Vision SDK、厂商 SDK。

### 3.4 前端

- Django Template。
- 原生 CSS。
- 少量原生 JavaScript。
- 不引入复杂前端框架。

## 4. 总体架构

系统采用分层架构：

```text
页面展示层
  Django Template / Views

业务服务层
  流程编排 / 生产记录 / MES交互 / 报警闭环 / 追溯查询

领域模型层
  产品 / 料框 / 配方 / 工序 / 视觉任务 / 报警 / 设备状态

设备适配层
  PLC / Robot / Scanner / Camera / LaserMarker / MES Adapter

视觉算法层
  料架定位 / 三轴偏差 / 配方校验 / 泡棉检测 / 标定参数

外部系统与现场设备
  PLC / 机器人 / MES / 相机 / 扫码枪 / 激光打标机
```

核心原则：

- Django 管业务状态，不抢 PLC 的实时控制职责。
- 设备通讯统一经过 adapter，业务代码不直接调用协议细节。
- 视觉算法统一放入 `vision.algorithms`，页面和流程服务只消费检测结果。
- 所有关键节点必须落库，便于追溯。
- 前期使用模拟设备跑通流程，后期替换真实通讯适配器。

## 6. Django App 职责划分

### 6.1 core

基础公共模块。

职责：

- 定义公共枚举。
- 定义公共异常。
- 定义时间、编号、状态转换等工具函数。
- 放置跨 app 共享的基础服务。

不应放入具体业务流程，避免变成杂物模块。

### 6.2 production

生产数据模块。

职责：

- 管理产品记录。
- 管理料框和料架信息。
- 管理装箱配方。
- 管理产品条码、料框码、批次号。
- 记录单件产品与料框的绑定关系。

主要页面：

- 产品列表。
- 料框列表。
- 配方管理。
- 批次记录。

### 6.3 workflow

流程编排模块。

职责：

- 管理单件产品的主流程状态。
- 管理阶段一、阶段二、阶段三的状态转换。
- 记录每个流程节点的开始时间、完成时间、结果和异常。
- 对接 `devices`、`vision`、`mes`、`alarms` 服务。

注意：

- 这里不直接写 PLC 协议代码。
- 这里不直接写 OpenCV 算法。
- 这里只调用清晰的服务接口。

### 6.4 devices

设备适配模块。

职责：

- 管理设备基础档案。
- 记录设备在线状态。
- 提供统一设备 adapter。
- 前期提供模拟设备 adapter。
- 后期替换为真实 PLC、机器人、扫码枪、相机和打标机通讯。

主要设备类型：

- PLC。
- 注塑取件机器人。
- 装箱机器人。
- 扫码枪。
- 深度相机。
- 固定检测相机。
- 激光打标机。
- 泡棉供料台距离传感器。

### 6.5 vision

视觉任务模块。

职责：

- 管理视觉采集任务。
- 保存原始图片、结果图片、检测数据。
- 执行或调用 OpenCV 算法。
- 记录料架定位结果。
- 记录泡棉检测结果。
- 管理手眼标定参数。

视觉模块只输出结构化结果，不直接决定整条产线流程。流程是否继续由 `workflow` 根据视觉结果和配方校验共同判断。

### 6.6 mes

MES 集成模块。

职责：

- 根据料框码获取装箱配方。
- 上传产品条码和料框码。
- 上传装箱完成记录。
- 上传视觉检测结果。
- 上传异常记录。
- 记录所有请求、响应、状态码和错误信息。

前期可实现模拟 MES client，返回固定配方和成功响应。

### 6.7 alarms

报警模块。

职责：

- 创建报警。
- 确认报警。
- 关闭报警。
- 记录人工处理意见。
- 支持工位锁定和解除。

报警来源包括设备通讯、扫码、MES、视觉检测、流程超时和配置校验失败。

### 6.8 traceability

追溯模块。

职责：

- 通过产品条码查询完整流程。
- 通过料框码查询装箱记录。
- 汇总产品、流程、视觉、MES、报警数据。
- 提供操作员查看页面。

追溯模块尽量不拥有复杂业务表，主要读取其他模块的数据并形成查询视图。

## 8. 流程状态设计

建议使用清晰的状态枚举，避免用中文散落在代码中。

### 8.1 主流程状态

```text
CREATED                 已创建
INJECTION_PICKED        注塑已取件
MARKING_READY           已到打标位
MARKED                  已打标
BARCODE_READ            条码已读取
MES_UPLOADED            MES已上传
HANDOVER_WAITING        待交接
HANDOVER_ROBOT_READY    装箱机器人已就绪
HANDOVER_GRIPPED        装箱机器人已抓牢
INJECTION_RELEASED      注塑机器人已释放
RACK_SCAN_READY         待料框扫码
RACK_SCANNED            料框已扫码
RECIPE_LOADED           配方已加载
RACK_LOCATING           料架定位中
RACK_LOCATED            料架定位完成
RECIPE_VERIFIED         配方校验通过
BOXING                  装箱中
FOAM_PICKING            泡棉抓取中
FOAM_ATTACHING          泡棉贴附中
FOAM_INSPECTING         泡棉检测中
COMPLETED               工序完成
LOCKED                  异常锁定
FAILED                  流程失败
```

### 8.2 状态转换原则

- 状态转换必须通过 `workflow.services` 完成。
- 每次状态变化必须写入 `WorkflowEvent`。
- 设备信号、视觉结果、MES 响应和人工操作都应作为事件来源。
- 失败状态应创建报警。
- `LOCKED` 状态只能由人工确认后解除。

## 9. 关键业务流程设计

### 9.1 打标与扫码流程

1. PLC 通知产品到达打标位。
2. 系统记录产品进入 `MARKING_READY`。
3. PLC 触发激光打标机。
4. 打标机完成后，扫码枪读取产品条码。
5. 系统记录 `BarcodeRecord`。
6. 系统调用 MES 上传产品条码和料框码。
7. MES 成功后，流程进入 `MES_UPLOADED`。
8. 任一环节失败时创建报警。

### 9.2 空中交接流程

1. 注塑机器人发送待交接信号。
2. 装箱机器人发送交接位就绪信号。
3. PLC 下发抓取指令。
4. 装箱机器人真空确认。
5. PLC 下发注塑机器人释放指令。
6. 注塑机器人释放完成。
7. 系统记录完整交接事件链。

### 9.3 料框扫码与配方获取流程

1. 装箱机器人携带读码器读取料框码。
2. 系统记录料框码。
3. 系统调用 MES 获取料架配方。
4. 本地保存或更新 `RackRecipe`。
5. 产品与料框建立绑定。
6. 配方获取失败时报警并锁定流程。

### 9.4 料架视觉定位流程

1. 装箱机器人移动到左侧料架拍照位。
2. 深度相机采集图片或点云。
3. OpenCV 算法计算左侧 X/Y/Z 偏差。
4. 装箱机器人移动到右侧料架拍照位。
5. 深度相机采集图片或点云。
6. OpenCV 算法计算右侧 X/Y/Z 偏差。
7. 系统保存左右料架定位结果。
8. 系统将补偿数据通过 PLC 下发给机器人。

### 9.5 配方校验流程

1. 视觉系统测量料架层高和层距。
2. 系统读取 MES 配方中的层高和层距。
3. 系统按容差进行比对。
4. 校验通过后向 PLC 输出可装箱信号。
5. 校验失败时报警并锁定工位。

### 9.6 装箱与泡棉贴附流程

1. PLC 将三轴补偿值下发给装箱机器人。
2. 装箱机器人按配方数量循环装箱。
3. 每次装箱后机器人移动到泡棉供料台。
4. 供料台距离传感器反馈泡棉高度。
5. 机器人调整抓取高度并吸取泡棉。
6. 机器人回到料架工位贴附泡棉。
7. 固定检测相机拍照。
8. OpenCV 检测泡棉是否存在、是否偏移、是否起翘。
9. 合格则进入下一循环。
10. 不合格则报警并锁定工位。

## 10. 设备适配器设计

### 10.1 适配器原则

所有设备访问必须经过统一 adapter，不允许在 view、model 或 workflow 中直接写协议代码。

建议定义基础接口：

```text
connect()
disconnect()
is_online()
read()
write()
execute()
```

不同设备可按需要扩展具体方法。

### 10.2 PLC Adapter

职责：

- 读取现场信号。
- 写入控制信号。
- 下发视觉补偿数据。
- 读取交接状态。
- 记录原始通讯数据。

典型方法：

```text
read_signal(signal_name)
write_signal(signal_name, value)
send_offsets(product_code, side, x, y, z)
send_workstation_lock(reason)
send_workstation_unlock()
```

### 10.3 Scanner Adapter

职责：

- 读取产品条码。
- 读取料框条码。
- 记录扫码失败。

典型方法：

```text
read_product_code()
read_rack_code()
```

### 10.4 Camera Adapter

职责：

- 触发拍照。
- 获取图片。
- 保存图片。
- 返回图片路径和采集元数据。

典型方法：

```text
capture(camera_code, task_type)
```

### 10.5 LaserMarker Adapter

职责：

- 接收或记录打标触发。
- 查询打标结果。
- 记录打标异常。

典型方法：

```text
trigger_mark(product_code)
get_mark_result()
```

### 10.6 Simulated Adapter

项目初期必须实现模拟适配器，用于在没有现场设备时跑通页面和流程。

模拟能力：

- 返回固定产品条码。
- 返回固定料框码。
- 返回固定 MES 配方。
- 返回固定视觉偏差。
- 返回泡棉检测合格或不合格结果。
- 模拟设备离线、扫码失败、配方校验失败等异常。

## 11. 视觉算法模块设计

### 11.1 料架定位 rack_locator

输入：

- 左侧或右侧相机图片。
- 标定参数。
- 料架配方。
- 料架定位基准配置。

输出：

- offset_x。
- offset_y。
- offset_z。
- measured_layer_height。
- measured_layer_spacing。
- confidence。
- is_success。
- debug_image_path。

前期实现建议：

- 先支持基于样例图片或模拟图片返回固定偏差。
- 再逐步接入真实深度图或点云。
- 算法结果必须结构化保存，不能只打印日志。

### 11.2 泡棉检测 foam_inspector

输入：

- 固定检测相机图片。
- 产品或料架位置索引。
- 检测区域配置。

输出：

- is_present。
- is_aligned。
- has_lifted_edge。
- score。
- is_passed。
- debug_image_path。

检测目标：

- 泡棉是否存在。
- 泡棉是否贴在指定区域。
- 泡棉是否明显偏移。
- 泡棉是否起翘。

前期实现建议：

- 先定义结果结构和图片归档。
- 初期可使用阈值、轮廓、模板匹配等基础方法。
- 后期如现场复杂，再考虑深度学习模型。

### 11.3 标定 calibration

职责：

- 保存相机与机器人之间的手眼标定参数。
- 保存坐标转换矩阵。
- 支持不同设备、不同工位的参数版本。

注意：

- 标定参数必须版本化。
- 视觉任务必须记录使用了哪个标定版本。

## 12. MES 集成设计

### 12.1 建议接口

前期按照抽象动作设计，不绑定具体 MES 协议。

```text
get_rack_recipe(rack_code)
upload_product_barcode(product_code, rack_code)

```

### 12.2 MES 交互记录

所有 MES 请求必须落库，至少保存：

- 请求动作。
- 请求参数。
- 响应内容。
- 成功状态。
- 错误信息。
- 请求时间。
- 响应时间。

### 12.3 失败处理

- MES 获取配方失败：流程锁定，提示人工处理。
- MES 上传条码失败：允许重试，超过次数后报警。
- MES 上传结果失败：保留本地记录，支持后续补传。

## 14. 后台任务与运行方式

项目初期可使用 Django management command 承担后台轮询任务。

建议命令：

```text
python manage.py run_device_polling
python manage.py run_workflow_worker
python manage.py run_mes_retry
```

职责：

- `run_device_polling`：轮询 PLC、扫码枪、相机等设备状态。
- `run_workflow_worker`：根据设备事件推进流程状态。
- `run_mes_retry`：补传失败的 MES 记录。

如果后期任务复杂、并发量增加，再引入 Celery 或独立服务。

## 17. 异常与报警策略

### 17.1 需要报警的场景

- 产品条码读取失败。
- 料框码读取失败。
- MES 配方获取失败。
- MES 上传失败超过重试次数。
- PLC 通讯中断。
- 机器人状态异常。
- 视觉定位失败。
- 料架配置校验失败。
- 泡棉检测不合格。
- 工序超时。
- 手动急停或工位锁定。

### 17.2 报警处理流程

1. 系统创建报警。
2. 根据报警等级决定是否锁定工位。
3. 页面显示当前报警。
4. 操作员确认报警。
5. 操作员填写处理备注。
6. 具备恢复条件后解除锁定。
7. 流程继续或转为失败。

## 23. 总结

本项目应优先把系统边界设计清楚：PLC 和机器人负责现场动作，Django 负责数据、流程、视觉结果、报警和追溯。第一阶段不要追求真实设备一步到位，应先使用模拟设备跑通完整业务闭环，再逐步替换真实通讯适配器。

这样的架构可以降低现场联调风险，也方便 Claude 后续按模块逐步实现业务代码。
